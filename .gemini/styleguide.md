# Gemini Code Assist Review Rules

- Always communicate in Chinese, including the review summary, inline comments, and suggested fixes.
- Use a Greptile-like high-confidence review standard: only report an issue when it is backed by clear evidence from the diff, existing code paths, tests, documentation, or reproducible commands. Mark uncertain findings as questions or risks, not blockers.
- In the review summary, explain what the PR changes, which modules are affected, whether it is safe to merge, and give a confidence score from 1 to 5. If the score is below 4/5, list the blocking reasons and the missing evidence.
- Use severity levels for inline comments. P1 means the change can cause build failures, runtime exceptions, configuration write errors, permission or security issues, data loss, or obvious user-visible regressions. P2 means test gaps, edge cases, compatibility risks, or paths likely to regress later. Do not mark pure style preferences as P1 or P2.
- Each finding should include the trigger condition, the real impact, why existing code or tests do not catch it, and a suggested fix direction. Provide a minimal suggestion only when it is clearly correct. Avoid broad or generic advice.
- When re-reviewing the same PR, first check whether previous findings were actually fixed. List fixed/addressed items explicitly. If a fix only addresses the symptom, explain the remaining path.
- Check whether the PR keeps its scope isolated: one PR should solve one clear problem. Flag unrelated refactors, formatting churn, or documentation rewrites that make the review riskier.
- For feature or bugfix changes, verify that tests cover the failing path. The main validation command for this repository is `pnpm test`, which runs `pnpm build && node --test tests/*.test.mjs`.
- For CLI behavior, tool configuration writes, locale files, version metadata, or README changes, require reproducible proof such as `node dist/cli.js --version`, `node dist/cli.js --help`, `node dist/cli.js doctor`, or `node dist/cli.js lang show`. Changes with no visible UI still need proof of fix through logs, command output, or test results.
- Check TypeScript and ESM constraints: import paths must include the `.js` extension, code must satisfy strict TypeScript, new code comments should be written in Chinese, and API response bodies should use English.
- Check i18n consistency: when user-visible text is added or changed, update both `src/locales/zh_CN.json` and `src/locales/en_US.json`, and preserve `{{param}}` template placeholders.
- Check whether tool integrations follow the existing architecture: new tools should implement `ITool`, be registered in `ToolManager`, and reuse the existing `configManager`, `i18n`, model service, and tool configuration write patterns.
- Prioritize these high-value checks for this repository: whether a new tool is fully registered in commands, menus, docs, and tests; whether configuration writes cover all target files while preserving user settings; whether API keys, endpoints, model names, and language settings can leak, disappear, or drift across paths; whether version numbers and release artifacts stay synchronized.
- For documentation, JSON, locale files, generated artifacts, or large mechanical changes, focus on consistency and missing fields rather than line-by-line wording. If generated files or `dist/` appear in a PR, verify that they are intended to be committed.
- Sort review output by severity and include file and line references. If no issues are found, say so clearly and mention any missing verification evidence.
